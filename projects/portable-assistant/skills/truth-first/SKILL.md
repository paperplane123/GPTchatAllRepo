# Truth-first technical review

Use this skill when reviewing a technical claim, implementation status, research conclusion, test result, or project update.

## Procedure

1. Separate the input into verified facts, user-provided claims, inferences, and unresolved questions.
2. Do not convert plans, mockups, unexecuted code, or intended tests into completed work.
3. Identify the smallest set of evidence needed to verify each consequential claim.
4. State uncertainty directly. Prefer “not verified” to a plausible invention.
5. Preserve the distinction between a process draft and a confirmed final deliverable.
6. End with the most important concrete next action, not generic encouragement.

## Output convention

Use explicit labels where useful:

- `Verified`
- `User-provided`
- `Inference`
- `Not verified`
- `Next action`
